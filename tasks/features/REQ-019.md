# User Story

As an approver or manager, I want PR packets to present the most important change and risk information visually so I can review faster and with less ambiguity.

# Goal

Define the reviewer-facing information architecture and visual language for PR packets before automation starts generating them at scale.

# Deliverables

- reviewer-facing packet section order
- table conventions for changed areas, validation status, and risk matrix
- rules for when Mermaid or SVG visuals are required
- guidance on what information belongs in prose versus structured tables
- example packet layout for engineering review use

# In Scope

- packet section ordering for review readability
- visual and table conventions
- reviewer checklist presentation guidance
- diagram usage guidance for architecture or workflow impact

# Out of Scope

- pixel-perfect UI implementation for packet viewing
- generic design system work outside the review packet use case

# Acceptance Criteria

1. Packet guidance defines a standard section order that prioritizes executive summary, changed areas, validation evidence, risks, and reviewer checklist.
2. The visual language specifies where tables are preferred over prose for reviewer efficiency, including changed areas and validation status.
3. Diagram guidance states when architecture-impact or control-flow changes require a Mermaid or SVG visual rather than text alone.
4. The layout guidance distinguishes mandatory reviewer-facing content from optional embellishment so generated packets remain concise.
5. At least one example or reference layout exists, is referenced from the relevant pattern or ADR documentation, and can be followed consistently by future packet generators.

# Validation Notes

- documentation review should confirm the packet layout is readable without opening every changed file
- example packet artifacts should be checked for table clarity and visual usefulness before automation is rolled out broadly

# Dependency Notes

- Depends on `REQ-018` because packet visuals and reviewer layout should be layered on top of an agreed packet contract rather than redefining it.
- This story should complete before merge-gating logic enforces packet completeness for architecture-impacting changes.
