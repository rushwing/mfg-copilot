# User Story

As an operations engineer, I want a grounded triage workflow so I can narrow likely causes faster during incidents.

# Goal

Provide the first operational troubleshooting workflow for build and MP operations.

# Deliverables

- ops-agent triage entry path
- correlated retrieval path across logs, metrics, and build history
- grounded triage summary output
- portal or API rendering for triage results

# In Scope

- log and metric correlation
- build and failure-history lookup
- triage summary generation

# Out of Scope

- automatic remediation

# Acceptance Criteria

1. Ops users can submit a build, MP, or environment issue and receive a grounded triage summary.
2. Outputs cite relevant logs, metrics, alerts, or historical records used in the reasoning path.
3. Likely causes, evidence, and recommended next actions are clearly separated in the result.
4. The workflow can handle partial or noisy data without failing silently.
5. Policy-driven model routing is applied for operational workloads.

# Validation Notes

- harness evals should include noisy, ambiguous, and partial-data scenarios

# Dependency Notes

- Depends on `REQ-004` for normalized read retrieval across operational evidence sources and on `REQ-005` because ops workloads may need distinct model-routing constraints.
- This story should not be blocked on execute-capable toolhub adapters when only read-only diagnostic inputs are required.
