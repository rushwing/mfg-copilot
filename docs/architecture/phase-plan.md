# Phase Plan

This document turns the current architecture scope into a delivery roadmap with explicit phase boundaries, dependencies, and exit criteria.

## Planning principles

- Sequence by business value and operational risk, not by architectural completeness.
- Ship read-heavy and approval-friendly workflows before execution-heavy automation.
- Establish traceability, schemas, and evaluation early so later automation does not outpace governance.
- Keep the first production path narrow and deeply instrumented.

## Phase 0: Foundation and Platform Readiness

### Goal

Create the platform foundation required for safe Phase 1 delivery.

### Scope

- repository structure, shared schemas, and ADR baselines
- React portal shell and FastAPI service skeletons
- PostgreSQL platform schema foundation
- KB ingestion path and RAG service contract shape
- LangSmith trace correlation contract
- model-routing schema and policy baseline
- test and eval framework setup
- CI baseline for lint, unit tests, and docs validation

### Deliverables

- working monorepo scaffolding and service boundaries
- approval schema scaffold
- retrieval schema scaffold
- phase planning, requirements, and design docs
- baseline developer workflow and CI definition

### Dependencies

- target environments for local dev and shared dev cluster
- initial SSO direction
- access to KB source repository and sample manufacturing data

### Exit criteria

- core docs, schemas, and repo conventions are accepted
- a developer can run portal and backend skeletons locally
- trace and eval conventions are decided
- platform team agrees on PostgreSQL ownership and MySQL integration strategy

## Phase 1: Doc Copilot and Read-Only Manufacturing Assistant

### Goal

Launch the first valuable workflow: documentation and checkpoint preparation with strong grounding, visibility, and low operational risk.

### Scope

- portal authentication and session context
- role-aware router for read-only workflows
- doc copilot MVP
- KB retrieval with citations
- project and product context retrieval
- template-driven structured generation
- gap detection and checklist generation
- daily / weekly / monthly summary generation
- read-only observability views and audit trail

### Included services

- `apps/portal-web`
- `services/api-gateway`
- `services/agent-router`
- `services/orchestrator`
- `services/agents/doc-copilot`
- `services/rag-service`
- minimal `services/toolhub` read-only adapters

### Key assumptions

- this phase does not execute manufacturing actions
- approvals may exist in the UI and schema but are informational or preview-only
- enterprise integrations are read-only

### Exit criteria

- users can ask documentation and checkpoint questions from the portal
- generated outputs include citations and template validation results
- summaries can be generated by project, phase, checkpoint, or station context
- traceability exists from user request to LangSmith run and audit log
- harness datasets cover core doc copilot scenarios

### Out of scope

- station execution
- software deployment
- write-back to enterprise systems
- autonomous remediation

## Phase 2: Deployment Planning and Controlled Execution

### Goal

Add deployment workflows with human approval and tightly scoped execution.

### Scope

- deploy agent MVP
- approval inbox and decision flow
- execution-safe orchestration state transitions
- MES / SFC / software distribution adapters
- deployment planning by site and station context
- IQ / OQ evidence generation
- audit-ready execution ledger

### Included capabilities

- plan generation with explicit pre-checks
- approval request creation and resolution
- execution only after approved state transitions
- evidence collection and exportable audit package

### Dependencies

- trusted identity and role mapping
- approval ownership model
- safe read / execute boundaries for enterprise tools
- validated toolhub adapters for target environments

### Exit criteria

- an approver can review and approve a deployment plan in the portal
- orchestrator enforces approval gating before execution
- deploy agent can run a narrow, pre-approved deployment flow for a selected station type
- execution results and IQ / OQ evidence are persisted and traceable
- harness and integration suites cover at least one approved execution workflow end to end

### Out of scope

- broad autonomous execution without human review
- unrestricted multi-site rollout
- advanced remediation during failed deployment

## Phase 3: Ops Agent and Multi-Agent Triage

### Goal

Add operational triage, diagnostics, and escalation support for build and MP operations.

### Scope

- ops agent MVP
- log, metric, build, and failure-history correlation
- RCA draft generation
- FA assist workflows
- stronger observability, cost controls, and guardrails
- optional bug-fix proposal workflow for selected software issues

### Included capabilities

- alert-driven or user-driven triage entry points
- multi-step ReAct flows across logs, metrics, tickets, and build data
- recommendation versus escalation decisioning
- summary outputs for operations, RCA, and FA consumers

### Dependencies

- reliable observability ingestion
- searchable historical incident and failure data
- mature model-routing policies for cost and safety
- stronger eval coverage for operational workflows

### Exit criteria

- ops users can submit a build or environment issue and receive grounded triage output
- the system can correlate at least logs, metrics, and build records in one flow
- RCA drafts and escalation summaries are reviewable in the portal
- operational workflows have dedicated harness evals and load tests

### Out of scope

- fully autonomous incident remediation
- broad code-change automation without explicit policy approval

## Cross-phase workstreams

### Security and governance

- SSO, RBAC, approval boundaries, audit, export control, and PII policy enforcement

### Data and retrieval

- KB ingestion, enterprise data access, vector indexing, search indexing, and citation quality

### Observability and evals

- LangSmith traces, Prometheus metrics, Elasticsearch-backed operational search, harness datasets, and DeepEval gates

### Delivery and environments

- Docker and Kubernetes deployment assets
- environment overlays for dev, staging, and prod
- release checklists and rollback runbooks

## Recommended milestone shape

- Milestone A: foundation and local dev loop
- Milestone B: doc copilot alpha
- Milestone C: doc copilot beta with summaries and reporting
- Milestone D: deployment planning and approval beta
- Milestone E: narrow deployment execution GA for selected station types
- Milestone F: ops triage beta

## Program risks to watch early

- identity and approval ownership ambiguity
- weak enterprise data quality or inconsistent keys across systems
- RAG quality issues due to poor chunking, metadata, or template alignment
- tool adapter safety boundaries arriving later than execution features
- missing eval datasets for realistic manufacturing workflows
